分组密码CBC padding Oracle攻击

Padding Oracle

padding Oracle Attack是针对CBC连接模式的攻击，和具体的加密算法无关(也就是说和是使用AES还是DES加密或其他的加密算法无关)

产生的原理

  • 如果密文解密成明文之后，填充位判断正确，而且经过业务逻辑代码的校验之后，也是正确，那么服务端会会返回200的状态码
  • 如果密文解密成明文之后，填充位判断正确，但是业务逻辑判断不通过，也就是说明文有问题，那么服务端会返回200或者300等状态码
  • 最后如果密文解密成明文之后，填充位判断不正确，会返回500等状态码

填充位是否正确会影响到状态码

进行padding Oracle攻击的时候，已知初始化向量是一个必须的条件

与CBC 翻转攻击不同的是，这里不清楚解密之后的明文，只知道初始化向量IV，密文以及解密是否成功

同时与服务器进行交互的过程中(发送加密数据到服务端，服务端判断发送的密文解密之后的填充位是否正确–根据状态码)

初始化向量

初始化向量一般在密文的头部，且初始化向量和密文的分组等长–8字节

攻击方式

0和数异或之后的结果都是数本身–中间值和0异或之后的结果也是中间值本身

以下以8位的分组密码进行举例(PKCS#5填充)

  • 补充：PKCS#5和PKCS#7填充方法的区别
    • PKCS#5在填充方面时PKCS#7的一个子集
    • PKCS#5只是对8字节填充，填充内容位0x01-0x08
    • PKCS#7不仅仅时对8字节的填充，他的BlockSize = 1-255字节
    • 所以PKCS#5 可以向上转换为PKCS#7，但是PKCS#7不一定可以转换到PKCS#5

如果解密出来的padding不正确(padding 的位数和padding 内容不一致)–那么http服务的status code 会返回500

根据以上的规则进行穷举：

将初始化向量设置为0，当最终解出来的明文最后一位不会再使程序报出填充位错误的时候(0x01-0x08)

  • 将初始化向量IV设置为0

  • 将IV的最后一位递增，然后提交，在0x00-0xFF的范围内，只会有一个异或middle的最后一位得到0x01，也就是正确的padding，这个时候服务器会正常解密–但是解密出来的内容不是原来的明文

    • middle中间值为经过AES解密之后尚未和IV的值)是固定的—最多尝试的次数是256次

    • Plaintext–明文

    • old_IV–已知的加密的明文

    • IV–构造的IV，攻击时输入的IV，在原本的加密中无任何实际意义，仅仅作用于密文攻击推测出中间值的过程

    • //原理
middle[最后一位] ^ IV[最后一位] == 0x01
middle[最后一位] = IV[最后一位] ^ 0x01
//由于正常的解密步骤是middle异或加密时使用的old_IV，因此：
middle[最后一位] ^ old_IV[最后一位] = Plaintext[最后一位]
Plaintext[最后一位] = middle[最后一位] ^ old_IV[最后一位]
